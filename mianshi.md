# RAG 架构设计面试版

## 1. 项目定位

这不是一个简单的“向量检索 + 大模型回答”项目，而是一套面向企业知识库场景设计的混合检索增强问答系统。

它的核心目标有四个：

- 提高召回率，避免用户换个说法就检索不到
- 提高回答准确率，尽量让回答建立在真实证据上
- 控制 token 成本，避免把大量低价值上下文直接塞给模型
- 保持系统可扩展、可观测、可降级，便于从本地原型演进到企业级落地

从面试角度讲，这套系统的价值不在于“调了一个模型接口”，而在于完整实现了：

**离线入库设计、在线检索编排、图谱增强、混合排序、成本控制、答案校验、观测评测、企业级后端扩展**。

---

## 2. 为什么不能只做基础版 RAG

基础版 RAG 一般是：

**文档切块 -> Embedding -> TopK 检索 -> 拼 prompt -> LLM 回答**

这种方案能快速跑通，但在企业知识库里很容易遇到几个问题：

### 2.1 用户问题表达不稳定

同一个知识点可能有多种说法，比如简称、别名、口语化表达、不同部门的术语习惯。

如果只依赖单次向量检索：
- 容易漏召回
- 语义相近但业务不相关的片段也可能被召回

### 2.2 术语、专名、章节名不一定适合纯向量检索

像制度名、接口名、公式名、概念名、章节标题这类问题，经常需要精确关键词匹配。

如果没有关键词召回：
- 很可能“意思差不多”的内容排在前面
- 真正精确命中的内容反而排不到前面

### 2.3 单一检索通道很难处理关系型问题

比如：
- 某个概念依赖什么前置知识
- 某个制度和哪个流程节点关联
- 某个问题应该优先参考哪类资料

这些都不是简单“语义最像”就能解决的，需要结构化关系。

### 2.4 多 chunk 直接拼给模型成本高、噪音大

如果把 TopK 全部塞给模型：
- token 开销高
- 模型容易混用多个片段内容
- 回答看起来流畅，但依据边界不清晰

### 2.5 没有评测和监控就无法持续优化

很多 RAG 项目只能“看起来能用”，但回答到底是不是基于证据、召回率有没有提升、缓存有没有命中，往往没人说得清。

所以我这套设计不是只追求“能回答”，而是追求：

- **找得更全**
- **排得更准**
- **答得更稳**
- **成本更低**
- **过程可解释**

## 3. 总体架构设计

整个系统可以概括为：

**知识接入层 -> 多索引构建层 -> 混合检索层 -> 生成控制层 -> 校验评测层 -> 可观测层**

### 3.1 六层架构

#### 第一层：知识接入层
负责处理原始知识文档，包括：
- PDF
- DOCX
- Markdown
- TXT

核心能力：
- 文本解析
- 元数据注入
- 文件类型统一抽象

这一层解决的问题是：
- 企业知识散落在不同格式文件中，无法直接检索
- 文本之外还需要抽取结构化信息，方便后续增强检索

#### 第二层：多索引构建层
同一份知识会被构建成三套能力：
- 向量索引
- 关键词索引
- 知识图谱索引

这一层解决的问题是：
- 单一索引无法覆盖所有问题类型
- 企业级问答需要“语义 + 关键词 + 关系”三种能力协同

#### 第三层：混合检索层
在线查询时，不直接走某一个检索器，而是统一调度：
- 查询扩展
- 查询路由
- 图谱召回
- 向量召回
- 关键词召回
- 融合排序
- 多样性控制
- 语义缓存

这一层解决的问题是：
- 提高召回率
- 提高排序准确率
- 减少重复检索成本

#### 第四层：生成控制层
不是把所有候选片段都交给模型，而是：
- 只保留少量高质量证据
- 对普通片段做预算裁剪
- 对图谱关系上下文做压缩
- 构造严格受控的 prompt

这一层解决的问题是：
- 降低 token 成本
- 减少噪音输入
- 保持答案引用边界清晰

#### 第五层：校验评测层
系统不仅生成答案，还会评估：
- 回答是否 grounded
- 回答和证据的重合度
- 检索结果是否命中 gold source

这一层解决的问题是：
- 避免回答脱离证据
- 让优化有量化依据

#### 第六层：可观测层
通过 Prometheus 指标和评测接口记录：
- 请求量
- 延迟
- 缓存命中率
- 最终候选数量
- 回答 grounding 分数

这一层解决的问题是：
- 让系统从“黑盒能跑”变成“白盒可运营”

## 4. 整个 RAG 的核心链路

### 4.1 离线入库链路

离线阶段做的是“把原始知识变成可检索系统”。

执行流程：

1. 读取知识目录中的文档
2. 解析文本内容
3. 注入 metadata
4. 文档切块
5. 生成 Embedding
6. 写入本地向量索引
7. 同步写入 Milvus
8. 同步写入 Elasticsearch
9. 构建 JSON 图谱
10. 按配置同步 Neo4j

这个阶段的关键思想是：

**一份知识，多路索引，多种检索能力同时准备好。**

### 4.2 在线问答链路

在线阶段做的是“针对一个问题，找到最可信的证据，再低成本生成回答”。

执行流程：

1. 用户提问
2. 判断是否为寒暄类输入
3. 对问题做查询扩展
4. 对问题做复杂度路由
5. 尝试命中语义缓存
6. 图谱检索，先找概念、关系和高优先级来源
7. 向量召回，优先 Milvus，失败则回退本地索引
8. 关键词召回，优先 Elasticsearch，失败则回退本地词法评分
9. 把三路候选统一做融合排序
10. 在 rerank window 内做多样性控制
11. 输出最终候选给工作流
12. 对上下文按 token 预算做裁剪和压缩
13. 选取 1 条主证据生成回答
14. 对回答做 grounding 校验
15. 返回答案、检索摘要、图谱摘要、验证结果

这个链路的核心原则是：

- 先尽量把相关内容找全
- 再尽量把真正高价值证据排到前面
- 最后用最小上下文把答案答准

## 5. 核心模块设计与职责

## 5.1 ingest.py：知识入库总入口

职责：
- 调度整个知识库构建流程
- 统一串联文档读取、向量入库、图谱构建

它解决的问题：
- 原始知识入库流程分散，不利于重建和刷新
- 企业场景下需要标准化、可重复执行的入库过程

面试里可以这样讲：
- 我把原来松散的文档处理流程收敛成统一入库入口，一次执行完成多索引构建，便于后续做增量更新、重建和运维管理。

## 5.2 loaders.py：多格式文档接入

职责：
- 解析 PDF、DOCX、Markdown、TXT
- 把多种格式统一转成 Document

它解决的问题：
- 企业知识源异构，无法直接进入下游检索
- 不同格式文件的抽取逻辑差异大，需要统一封装

为什么重要：
- 如果接入层做不好，下游召回再强也无意义，因为原始知识没有被正确结构化

## 5.3 metadata_registry.py：结构化元数据中心

职责：
- 统一管理 subject、chapter、summary、concepts、aliases、relations 等元数据

它解决的问题：
- 纯文本检索对业务结构理解弱
- 企业知识需要“文本 + 结构化标签”共同参与召回和排序

为什么重要：
- 没有元数据，图谱构建、查询扩展、路由和结果解释都会很弱

## 5.4 vector_store.py：向量主通道

职责：
- 文档切块
- Embedding 生成
- 本地向量索引持久化
- 在线向量检索

它解决的问题：
- 语义相近但表述不同的问题无法靠关键词完全覆盖
- 企业知识中大量自然语言问法需要 semantic retrieval

为什么保留本地索引：
- 本地索引适合开发和小规模验证
- 即使 Milvus 不可用，系统也能跑通

面试亮点：
- 这不是把向量库做成唯一依赖，而是设计成主通道 + 回退通道，保证系统鲁棒性

## 5.5 milvus_store.py：企业级向量库

职责：
- 管理 collection
- 向量写入
- ANN 检索

它解决的问题：
- 本地 numpy 索引适合原型验证，不适合更大规模知识库
- 企业环境需要更好的向量存储能力、查询性能和扩展能力

为什么用 Milvus：
- 适合承接大规模 chunk 的语义召回
- 便于后续做索引参数调优、分区、扩容

面试说法：
- Milvus 不是为了“技术堆料”，而是解决本地索引无法支撑更大规模生产检索的问题

## 5.6 elastic_store.py：关键词召回通道

职责：
- 建索引
- 同步 chunk 到 ES
- 做 multi_match 词法检索

它解决的问题：
- 纯向量检索对术语、制度名、专有名词、章节标题类问题不稳定
- 企业问答里很多问题本质上是“精确术语命中”

为什么要和向量一起用：
- 向量擅长语义相似
- ES 擅长精确关键词和字段搜索
- 两者是互补关系，不是替代关系

面试里建议直接说：
- Elasticsearch 解决的是“用户问得不一定像文档，但又经常包含关键术语”的问题

## 5.7 graph_store.py：Graph RAG 核心

职责：
- 构建图谱
- JSON 持久化
- Neo4j 同步
- 图谱检索

它解决的问题：
- 单纯语义相似无法稳定表达概念之间的结构关系
- 系统需要知道“这个问题涉及哪些概念、这些概念之间有什么关系、优先参考哪些资料”

这里的关键不是“有图谱”，而是：
- 图谱真正参与了检索
- 图谱能输出概念命中、关系扩展、来源优先级
- 图谱结果还能生成 graph document 参与最终生成

面试深度点：
- 很多项目只是把知识图谱做成可视化展示，我这里把它变成了检索链路的一部分，这才是真正的 Graph RAG

## 5.8 retrieval_optimizer.py：策略层

职责：
- 查询分类
- 查询扩展
- token 预算控制
- 本地词法评分
- 语义缓存
- grounding 校验

它解决的问题：
- 所有问题都走相同检索深度，成本高且效果不稳定
- 查询别名、简称、同义词无法自然命中
- prompt 上下文过长导致 token 浪费
- 相似问题重复检索成本高
- 回答是否真正基于证据缺乏判断依据

为什么重要：
- 这一层本质上是把 RAG 从“检索调用”升级成“检索策略系统”

面试里可以突出：
- 我不仅做了检索器，还做了查询理解、预算治理和质量校验，这些是企业级 RAG 非常关键的部分

## 5.9 hybrid_retriever.py：在线检索编排层

职责：
- 编排图谱、向量、关键词三路召回
- 统一做打分融合
- 做结果多样性控制
- 输出检索摘要

它解决的问题：
- 多个检索器各自有价值，但单独使用都不够稳定
- 需要一个统一调度层把多路候选合并成最终结果集

它的核心价值：
- 不是只返回 TopK，而是做“候选构建 + 多因子融合 + rerank + diversify”

面试里可以说：
- 这是整个系统最核心的编排层，相当于检索侧的决策引擎

## 5.10 kb_match.py：相关性过滤层

职责：
- 判断某个 chunk 是否真正相关

它解决的问题：
- 检索召回出来的内容未必都适合进入最终生成阶段
- 如果不做二次过滤，模型很容易被弱相关内容干扰

## 5.11 graph/nodes.py：问答工作流节点

职责：
- 调起混合检索
- 过滤候选 chunk
- 压缩上下文
- 调用 LLM 生成答案
- 记录指标与校验结果

它解决的问题：
- 检索结果如何有序进入生成阶段
- 如何在工作流中区分寒暄、知识库问答、兜底直答

为什么关键：
- 这里把检索层和生成层真正连成了一个可控工作流

## 5.12 evaluation.py：评测层

职责：
- 做检索评测
- 做回答 grounding 评测

它解决的问题：
- RAG 很容易停留在“主观感觉还行”
- 如果没有 Hit@K、MRR、grounding 等指标，就很难优化

面试建议：
- 可以强调自己不仅关注回答效果，还关注如何建立离线验证闭环

## 5.13 observability.py：可观测层

职责：
- 定义 Prometheus 指标
- 记录 HTTP、检索、回答质量、评测运行等指标

它解决的问题：
- RAG 系统上线后，如果没有观测能力，就无法知道瓶颈在哪、优化有没有效果

为什么要做 no-op fallback：
- 避免因为 `prometheus_client` 未安装导致整个应用不可用
- 企业增强能力不能破坏系统基本可用性

## 6. 每个技术解决了什么问题

这一部分面试特别重要，因为面试官往往不关心你“用了什么”，更关心你“为什么用”。

## 6.1 OpenAI Embeddings / 向量化模型

解决的问题：
- 用户问题和知识片段用词不一致时，关键词匹配容易失效
- 需要通过语义空间表达文本相似性

它的定位：
- 语义召回基础设施

## 6.2 RecursiveCharacterTextSplitter

解决的问题：
- 原始文档太长，不能直接进入检索和生成
- chunk 太大影响精度，chunk 太小又丢语义

它的定位：
- 检索粒度控制器

## 6.3 本地向量索引（documents.json + embeddings.npy）

解决的问题：
- 项目早期需要低成本验证 RAG 主流程
- 外部向量库不一定随时可用

它的定位：
- 本地默认索引和回退方案

## 6.4 Milvus

解决的问题：
- 本地索引无法支撑更大规模知识库和更高并发检索
- 企业场景需要专业向量数据库

它的定位：
- 企业级向量召回主通道

## 6.5 Elasticsearch

解决的问题：
- 专有名词、术语、章节名、摘要字段不适合只靠向量检索
- 需要字段级加权和精确文本匹配能力

它的定位：
- 关键词召回通道

## 6.6 Neo4j

解决的问题：
- 复杂问题中的概念关系、上下游依赖、来源优先级无法靠纯文本很好表达
- 需要真实图数据库做概念和关系检索

它的定位：
- 图谱增强检索通道

## 6.7 JSON 图谱

解决的问题：
- 并不是所有环境都能部署 Neo4j
- 需要一个轻量、可直接运行的图谱 fallback

它的定位：
- 轻量图谱运行模式

## 6.8 元数据 registry

解决的问题：
- 知识库如果只有正文，没有结构化标签，就很难做高质量路由、解释和排序

它的定位：
- 文本检索和结构化检索之间的桥梁

## 6.9 Query Expansion

解决的问题：
- 同义词、别名、简称导致的漏召回

它的定位：
- 提升召回率的前置增强层

## 6.10 Retrieval Router

解决的问题：
- 简单问题和复杂问题如果走同一套召回深度，会浪费成本或召回不足

它的定位：
- 检索资源调度器

## 6.11 RRF 融合

解决的问题：
- 多路召回分值不在同一量纲，直接相加不稳定
- 某一路分值很大时可能完全压制其他路径

它的定位：
- 多路检索结果统一融合机制

## 6.12 Diversity / Max Per Source

解决的问题：
- 同一 source 下相似 chunk 过多，占满候选窗口

它的定位：
- 最终候选结果去冗余机制

## 6.13 Token Budget Control

解决的问题：
- prompt 过长导致成本高、噪音大、回答不稳定

它的定位：
- 生成阶段的成本治理能力

## 6.14 Semantic Cache

解决的问题：
- 相似问题反复检索，延迟高、重复计算多

它的定位：
- 检索阶段的性能优化能力

## 6.15 Grounding Validation

解决的问题：
- 模型回答看起来合理，但不一定真正基于证据

它的定位：
- 回答质量与可信度校验器

## 6.16 Prometheus

解决的问题：
- 上线后无法量化知道检索效果、延迟、缓存命中、回答质量变化

它的定位：
- 系统运行观测与优化依据

## 7. 为什么这套设计有“深度”

如果面试官问“你的 RAG 为什么算有深度”，我建议从下面几个层次回答。

### 7.1 不是单一检索，而是混合检索

从基础 RAG 的单向量召回，升级成：
- 向量召回
- 关键词召回
- 图谱召回

这说明系统已经不是 Demo，而是在认真解决召回率和准确率问题。

### 7.2 不是只会查，还会做检索策略

通过：
- 查询扩展
- 路由分类
- 语义缓存
- 多样性控制

说明系统已经具备“按问题类型调检索策略”的能力。

### 7.3 不是只关心回答，还关心成本

通过：
- chunk 预算控制
- graph 预算控制
- 单证据生成策略

说明系统考虑了实际线上成本，而不是只追求堆上下文。

### 7.4 不是只追求能答，还关心可信度

通过：
- grounding 校验
- 检索摘要
- 评测接口

说明系统具备质量闭环，不是黑盒输出。

### 7.5 不是强依赖某个组件，而是支持回退运行

比如：
- Milvus 不可用可回退本地索引
- Elasticsearch 不可用可回退本地词法评分
- Neo4j 不可用可回退 JSON 图谱
- Prometheus 不可用可回退 no-op 指标

这说明工程设计考虑了稳定性，不是实验室式实现。

## 8. 从面试视角怎么讲这套架构

## 8.1 一句话版本

我做的不是基础版向量 RAG，而是一套带图谱增强、关键词召回、向量召回、查询扩展、缓存、成本控制和观测评测能力的企业级混合检索 RAG。

## 8.2 1 分钟版本

我这套 RAG 系统在架构上分成离线入库和在线问答两条主链路。离线阶段会把 PDF、DOCX、Markdown 等知识文档统一解析、切块和向量化，同时构建本地索引、Milvus 向量库、Elasticsearch 关键词索引，以及 JSON 和 Neo4j 双后端知识图谱。在线查询时，系统不会只走一个检索器，而是先做查询扩展和问题路由，再优先查语义缓存；如果没命中，就先用图谱识别概念、关系和高优先级来源，再分别做向量召回和关键词召回，最后用多因子融合排序把三路结果合成最终候选。生成阶段我会控制 token 预算，只保留少量高质量证据，并做 grounding 校验和 Prometheus 指标埋点。所以这套系统解决的不只是“能回答”，而是企业知识库里更重要的几个问题：召回率、准确率、成本和可运营性。

## 8.3 2 分钟版本

如果展开讲，我的设计重点有三层。第一层是多索引构建，我不会只建向量索引，而是把同一份知识同时构建成向量索引、关键词索引和图谱索引，因为企业问题既有语义相似检索，也有精确术语检索，还有概念关系检索。第二层是混合检索编排，我做了一个统一的 hybrid retriever，在查询阶段先做查询扩展、复杂度路由和缓存判断，再分别执行图谱、向量和关键词三路召回，最后通过 graph、vector、lexical、coverage、intent 和 RRF 等信号做融合排序，并控制单一来源不要占满候选窗口。第三层是生成控制和质量闭环，我不会把很多 chunk 直接塞给模型，而是按 token 预算压缩上下文，只保留高质量主证据，再结合图谱关系生成答案，并通过 grounding 校验、评测接口和 Prometheus 指标去持续优化。所以从工程角度看，这套系统已经不只是 RAG Demo，而是一套具备企业级雏形的检索增强问答架构。

## 9. 面试官常问的深度问题

## 9.1 为什么要同时用 Elasticsearch、Milvus、Neo4j？

回答思路：
- Milvus 解决的是大规模语义召回问题
- Elasticsearch 解决的是精确术语和字段匹配问题
- Neo4j 解决的是概念关系和来源增强问题

一句话总结：
- 三者不是重复建设，而是分别解决语义、关键词、关系三类不同问题

## 9.2 为什么不直接把所有召回结果喂给大模型？

回答思路：
- token 成本高
- 噪音多
- 模型可能混用多段片段内容
- 答案依据边界不清楚

所以我做了：
- rerank
- 多样性控制
- token budget
- 单证据主回答策略

## 9.3 为什么要做图谱，不只做向量？

回答思路：
- 向量擅长语义相似，不擅长显式表达概念之间的结构关系
- 图谱能识别概念、扩展关系、提升来源优先级
- 对复杂问题、依赖关系问题、跨概念问题更有帮助

## 9.4 为什么要做缓存？

回答思路：
- 企业系统里高频问题很多，重复检索浪费资源
- 语义缓存可以降低延迟和成本
- 同时不会破坏回答质量，因为仍然有 route_type 和相似度阈值控制

## 9.5 为什么要做 grounding 校验？

回答思路：
- RAG 的核心不是“说得像”，而是“是否基于证据”
- grounding 校验可以帮助判断回答是不是脱离了参考内容
- 后续也可以把它作为离线评测和线上监控指标

## 11. 最终结论

这套 RAG 的架构设计，本质上是在回答一个企业级问题：

**如何让知识库问答既找得到、排得准、答得稳，还能控制成本并持续优化。**

我的答案不是只接一个向量库，而是通过：

- 多格式知识接入
- 结构化 metadata 建模
- 向量 + 关键词 + 图谱三路召回
- 查询扩展与路由
- 融合排序与多样性控制
- token 预算治理
- grounding 校验
- Prometheus 观测和评测接口

把整个链路做成一套可以演进、可以解释、可以优化的企业级 RAG 架构。

如果在面试中收尾，我会用一句话总结：

**我做的不只是“把文档喂给大模型”，而是把知识接入、检索增强、生成控制、质量校验和系统观测串成了一整套可落地的 RAG 工程体系。**
