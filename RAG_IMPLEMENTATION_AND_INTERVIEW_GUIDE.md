# RAG 实现文档与面试简要话术

## 1. 项目中 RAG 的实现目标

本项目实现的是一套面向教育场景的轻量 Graph RAG 系统，目标是：

- 支持多格式知识文件入库
- 支持向量检索与知识图谱增强检索
- 支持人工单选证据片段后再生成答案
- 提升回答的可控性、准确率和可解释性

当前实现不是单纯的“embedding + topk + prompt”，而是：

**文档解析 → 元数据增强 → 向量索引 → 图谱索引 → 混合检索 → 相关性过滤 → 用户选片段 → 大模型生成答案**

## 2. 代码实现分层

### 2.1 配置层

文件：
- `backend/app/config.py`

作用：
- 统一管理知识目录、向量索引目录、图谱索引目录、静态目录、元数据文件路径等配置

当前关键配置包括：
- `knowledge_dir`
- `vector_index_dir`
- `graph_index_dir`
- `knowledge_metadata_file`

---

### 2.2 数据源与元数据层

文件：
- `backend/data/knowledge/`
- `backend/data/knowledge_metadata.json`
- `backend/app/rag/metadata_registry.py`

作用：
- `knowledge/` 存放原始知识文件
- `knowledge_metadata.json` 存放结构化元数据
- `metadata_registry.py` 负责统一读取 source、concept、relation 信息

元数据主要包括：
- 学科 `subject`
- 年级 `grade`
- 章节 `chapter`
- 难度 `difficulty`
- 摘要 `summary`
- 概念 `concepts`
- 别名 `aliases`
- 关系 `relations`

这样做的目的，是让检索不只依赖纯文本，而是具备结构化语义。

---

### 2.3 文档加载层

文件：
- `backend/app/rag/loaders.py`

作用：
- 加载 PDF、DOCX、Markdown、TXT 等知识文件
- 将文档统一转成 `Document`
- 给每个文档注入 metadata

处理方式：
- PDF：通过 `pypdf` 提取文本
- DOCX：通过 `python-docx` 提取段落
- Markdown/TXT：直接读取文本
- 读取后统一注入：
  - `source`
  - `file_type`
  - `subject`
  - `chapter`
  - `difficulty`
  - `concepts`
  - `aliases`

---

### 2.4 向量索引层

文件：
- `backend/app/rag/vector_store.py`

作用：
- 对文档切块
- 生成 embedding
- 存储本地向量索引
- 执行相似度检索

核心实现：
- 使用 `RecursiveCharacterTextSplitter` 切块
- 使用 `OpenAIEmbeddings` 对 chunk 向量化
- 使用 `numpy` 保存向量矩阵
- 使用 `documents.json` 保存文本和 metadata
- 检索时对 query 做 embedding，再进行向量相似度排序

这层实现的是项目最基础的语义召回能力。

---

### 2.5 图谱索引层

文件：
- `backend/app/rag/graph_store.py`

作用：
- 构建轻量知识图谱
- 存储概念节点、来源节点、关系边
- 提供图谱概览和图谱搜索能力

图谱内容主要包括：
- 概念节点 `concepts`
- 来源节点 `sources`
- 关系边 `relations`

例如：
- 导数 → 前置依赖 → 极限
- 牛顿第二定律 → 表达式 → F=ma
- 月亮意象 → 常表达 → 思乡

图谱检索的价值在于：
- 识别问题中涉及的核心概念
- 找到关联概念
- 关联高质量知识来源
- 对向量检索结果进行增强和重排

---

### 2.6 混合检索层

文件：
- `backend/app/rag/hybrid_retriever.py`
- `backend/app/rag/kb_match.py`

作用：
- 融合图谱召回和向量召回
- 对结果做 boost 和过滤
- 提高召回率与准确率

实现方式：
1. 先通过图谱识别问题命中的概念
2. 通过图谱给知识来源打分
3. 再执行向量检索
4. 把图谱分数作为 boost 作用到向量检索结果上
5. 最后用关键词重叠、概念重叠、别名匹配做过滤

这样可以避免单纯语义相似但业务上不相关的问题。

---

### 2.7 入库层

文件：
- `backend/app/rag/ingest.py`

作用：
- 执行知识入库
- 同时构建向量索引和图谱索引
- 返回知识库统计结果

入库流程：
1. 从知识目录读取文档
2. 注入 metadata
3. 切块并做 embedding
4. 构建向量索引
5. 构建图谱索引
6. 返回 chunks、files、subjects、graph stats 等信息

---

### 2.8 问答工作流层

文件：
- `backend/app/graph/workflow.py`
- `backend/app/graph/nodes.py`
- `backend/app/graph/state.py`

作用：
- 编排用户问答流程
- 管理状态流转
- 控制知识检索和答案生成

当前流程包括：
1. 判断是否是打招呼
2. 如果不是，进入混合检索
3. 返回候选知识片段给前端
4. 用户单选一条知识片段
5. 系统把问题、片段、图谱上下文一起交给大模型生成答案
6. 如果知识库无命中，则直接让大模型回答

这套流程的优点是：
- 比直接 topk 喂给模型更可控
- 用户可参与证据选择
- 回答可解释性更强

## 3. 当前 RAG 的完整实现流程

### 第一步：准备知识数据

把 PDF、DOCX、Markdown、TXT 放入知识目录，再通过 `knowledge_metadata.json` 补充结构化元数据。

### 第二步：执行入库

系统读取所有知识文件，加载文本后注入 metadata，然后切块并生成 embedding，同时构建图谱索引。

### 第三步：用户提问

用户发起问题后，系统先判断是不是简单问候，如果不是，就进入检索流程。

### 第四步：图谱检索

先从图谱中识别概念、别名和关系，得到：
- 命中概念
- 关联概念
- 推荐来源

### 第五步：向量检索

再从向量索引中召回语义最接近的文本片段。

### 第六步：混合排序与过滤

将图谱结果和向量结果融合，并通过关键词、概念、别名等规则过滤结果。

### 第七步：用户选择证据

前端把候选知识片段展示给用户，用户单选 1 条作为回答依据。

### 第八步：模型生成答案

系统将：
- 用户问题
- 所选知识片段
- 图谱补充关系

组合进 prompt，由大模型生成最终回答。

## 4. 这套实现的亮点

### 4.1 不只是基础向量 RAG

当前实现已经从单纯的向量检索升级为轻量 Graph RAG。

### 4.2 引入了结构化元数据

支持：
- 学科
- 章节
- 难度
- 概念
- 别名
- 关系

### 4.3 图谱增强检索

图谱不是单独存在，而是参与到检索排序中，对向量召回做增强。

### 4.4 回答过程可控

不是直接把多个 chunk 全喂给模型，而是让用户先选证据，再生成答案。

### 4.5 适合教育场景

教育场景天然存在：
- 概念关系
- 前置依赖
- 章节结构
- 知识点解释

所以 Graph RAG 的收益会更明显。

---

## 5. 当前方案的限制

### 5.1 图谱主要依赖手工元数据

当前图谱关系主要来源于 `knowledge_metadata.json`，还没有做自动抽取。

### 5.2 还没有引入 rerank

目前主要依靠图谱 boost + 规则过滤，未来可以再加 reranker 提升前排准确率。

### 5.3 还没有引入 BM25

如果后续公司数据量变大，建议加入关键词检索，形成真正的企业级混合检索。

## 8. 2 分钟面试口语稿

你可以按下面这段直接表达：

我这个项目实现的是一套面向教育场景的轻量 Graph RAG。最开始我不是只想做一个简单的“向量检索 + 大模型生成”，而是希望回答既准确，又能解释为什么这样答，所以我把整个链路拆成了知识入库、结构化增强、混合检索和生成四层。

在入库阶段，我先把 PDF、Word、Markdown、TXT 等不同格式的知识文件统一解析成文本，再通过切块策略把长文档拆成适合检索的 chunk。然后我调用 embedding 模型，把每个 chunk 转成向量，存到本地向量索引里，这样系统就具备了基本的语义召回能力。

但我发现如果只依赖向量检索，对于教育场景里很多“概念关系、前置知识、公式联系”类问题不够稳定，比如“导数为什么依赖极限”“牛顿第二定律和加速度是什么关系”，单纯语义相似不一定能稳定命中。所以我又补了一层结构化知识能力：给每个知识源增加学科、章节、难度、概念、别名和关系等 metadata，并基于这些 metadata 构建了一个轻量知识图谱。

最后在生成阶段，我没有直接把多个 chunk 全部喂给模型，而是让用户先单选一条证据片段，系统再把用户问题、所选片段和图谱补充关系一起交给大模型生成答案。这样做的好处是回答更可控，可解释性更强，也更适合教育类问答。

如果后续知识量继续增大，我会再往企业级方向升级，比如加入 BM25 混合检索、rerank、分层索引、权限过滤，以及从手工图谱演进到自动知识抽取。

## 10. 面试官高频追问 Q&A

### Q1：你这个 RAG 和普通 RAG 有什么区别？

答：
我这个项目不是纯向量 RAG，而是轻量 Graph RAG。普通 RAG 主要是文档切块、embedding、检索、生成；我在这个基础上增加了 metadata 和知识图谱，把概念、别名、关系也纳入检索流程。这样系统不仅能做语义召回，还能处理概念关系、前置知识、公式联系等更结构化的问题。

### Q2：为什么要做图谱增强，而不是只做向量检索？

答：
因为纯向量检索对于“关系型问题”不够稳定。比如“导数依赖什么”“牛顿第二定律和加速度的关系是什么”，这些问题只靠语义相似度容易召回到解释导数或解释加速度的文本，但不一定能召回两者之间关系最强的证据。图谱增强以后，可以先识别问题涉及的概念，再扩展相关关系和来源，对召回结果做 boost，准确率会更高。

### Q3：你的入库流程是怎么做的？

答：
入库主要分四步。第一步是多格式文档解析，把 PDF、DOCX、Markdown、TXT 统一转成文本对象；第二步是 metadata 注入，把 source、file_type、subject、chapter、concepts 这些结构化信息打到文档上；第三步是切块和 embedding，把长文本拆分成 chunk 并转成向量；第四步是同时构建向量索引和图谱索引，形成检索所需的两套基础能力。

### Q4：为什么没有直接上 FAISS、Milvus 或 Neo4j？

答：
这个项目现阶段的目标是先快速验证教育场景下 Graph RAG 的效果，所以我优先实现了轻量、可控、低依赖的方案。向量层先用 numpy + json 实现本地索引，图谱层先用 JSON 结构实现轻量知识图谱，这样便于理解全链路和快速调试。如果后续数据规模继续扩大，我会把向量层升级到专业向量库，把图谱层迁移到图数据库。

### Q5：你怎么提升召回率和准确率？

答：
召回率主要靠更好的索引和多路召回，比如 metadata、图谱概念扩展、向量检索、后续可加 BM25；准确率主要靠结果过滤和重排，比如关键词重叠、概念重叠、别名匹配，以及后续可加 rerank。当前项目已经先做了图谱 boost 和规则过滤，这一步明显优于单纯 TopK 向量召回。

### Q6：为什么让用户单选证据片段？

答：
因为我希望答案更可控。很多 RAG 系统会直接把 TopK 多个片段都交给模型，但这样模型可能会混用多段内容，回答虽然流畅但依据不清晰。让用户单选证据片段，相当于把最终证据锚定下来，回答可解释性更强，也更适合教育场景。

### Q7：如果公司知识量变大，你怎么升级？

答：
我会往企业级架构演进：先做数据清洗和 metadata 标准化，再加 BM25 + 向量的混合检索和 rerank，然后做分层索引、权限过滤和增量更新。如果结构化关系越来越复杂，再把现在的轻量图谱升级成自动知识抽取 + 图数据库方案。

### Q8：你在这个项目中最核心的贡献是什么？

答：
我最核心的贡献是把整个 RAG 从“能跑”升级成“可解释、可扩展”的系统：一方面把文档入库、向量索引、图谱索引、混合检索和生成串成完整闭环；另一方面针对教育场景增加了概念、关系和人工证据选择机制，让它不只是一个简单问答机器人，而是具备结构化知识能力的学习助手。

## 11. 最短背诵版

### 11.1 30 秒版

我这个项目实现的是一套轻量 Graph RAG。先把 PDF、Word、Markdown 等文档解析、切块并向量化，建立本地向量索引；再通过元数据构建概念和关系图谱；查询时结合图谱和向量做混合检索，再把命中的证据片段交给大模型生成答案。相比纯向量 RAG，这套方案更适合教育场景下的概念关系类问题。

### 11.2 15 秒版

我做的不是普通 RAG，而是轻量 Graph RAG：文档先向量化入库，再通过元数据构建知识图谱，查询时用图谱增强向量检索，最后结合证据片段生成更可控的答案。
