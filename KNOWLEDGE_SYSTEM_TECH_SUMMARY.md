# 知识体系技术总结

## 1. 总体定位

当前项目的知识体系已经从基础的向量 RAG，升级为一套轻量级的 Graph RAG 方案。整体目标是：

- 支持多格式知识文件入库
- 支持向量检索与结构化知识检索并存
- 支持概念、关系、章节、学科等元数据增强
- 支持在问答阶段把图谱关系和原始知识片段一起提供给大模型

这套方案没有引入重型图数据库，而是采用“本地向量索引 + 本地 JSON 图谱索引 + 元数据注册表”的实现方式，适合当前仓库规模和开发阶段。

---

## 2. 当前用到的核心技术

### 2.1 Web 与服务层

- **FastAPI**
  - 用于提供后端 API 服务
  - 负责健康检查、知识入库、聊天问答、图谱查询等接口
- **Pydantic / pydantic-settings**
  - 用于配置管理和请求模型定义
  - 统一管理知识目录、向量索引目录、图谱索引目录等配置

### 2.2 知识编排与问答流程

- **LangGraph**
  - 用于编排问答流程节点
  - 当前流程包括：打招呼判断、知识检索、人工单选、答案生成、知识库未命中时直接大模型回答
- **LangChain Core**
  - 使用 `Document` 作为知识载体
  - 使用消息对象组织大模型提示与对话内容

### 2.3 大模型与向量能力

- **OpenAI Chat 模型**
  - 用于最终答案生成、问候回复等
- **OpenAI Embeddings**
  - 用于知识片段向量化和查询向量化

### 2.4 本地向量检索

- **NumPy**
  - 用于保存和计算向量矩阵
  - 当前向量库没有使用 FAISS / Chroma，而是自定义实现了轻量本地向量索引
- **JSON 持久化**
  - 保存 chunk 文本与 metadata
- **余弦相似度检索**
  - 通过向量归一化后用点积得到相似度排序结果

### 2.5 文档切分与知识预处理

- **RecursiveCharacterTextSplitter**
  - 用于知识文档切块
  - 当前 chunk_size / overlap 已在项目中配置
- **多格式文档加载**
  - `pypdf`：读取 PDF
  - `python-docx`：读取 Word 文档
  - 原生文本读取：处理 Markdown / TXT

### 2.6 元数据与知识图谱

- **JSON 元数据注册表**
  - 通过 `knowledge_metadata.json` 维护每个知识源的结构化元数据
  - 包括：学科、年级、章节、难度、摘要、概念列表
- **轻量知识图谱索引**
  - 通过本地 JSON 存储概念节点、来源节点、关系边
  - 支持概念别名、概念关系、证据来源
- **Graph RAG**
  - 检索问题时先查图谱，再结合向量检索做混合召回
  - 用图谱结果提升向量检索排序质量，并补充结构化上下文

### 2.7 前端展示

- **React + TypeScript**
  - 用于问答界面和知识片段选择界面
- **Vite**
  - 用于前端开发与构建
- **类型化 API 数据结构**
  - 前端已支持展示图谱命中的概念、关联概念、chunk 元数据

### 2.8 部署与容器化

- **Docker**
  - 用于容器化运行后端与前端开发服务
- **SSH 插件容器入口**
  - 便于在容器内进行远程连接与调试

## 3. 当前知识体系的分层结构

### 3.1 原始知识层

来源于 `backend/data/knowledge/` 目录中的：

- PDF
- DOCX
- Markdown
- TXT

### 3.2 元数据层

来源于 `backend/data/knowledge_metadata.json`，为知识源补充：

- subject
- grade
- chapter
- difficulty
- summary
- concepts
- aliases
- relations

### 3.3 向量索引层

把文档切块后进行 embedding，生成：

- `documents.json`
- `embeddings.npy`

用于语义检索。

### 3.4 图谱索引层

把概念、来源、关系边构造成图谱，生成：

- `graph.json`

用于概念识别、关系扩展、知识源加权。

### 3.5 混合检索层

综合：

- 图谱命中概念
- 图谱关系边
- 图谱推荐来源
- 向量相似度结果
- 关键词/概念重叠判断

最终筛出最适合展示给用户的知识片段。

### 3.6 答案生成层

大模型接收：

- 用户问题
- 用户选择的知识片段
- 图谱补充关系（如果命中）

再生成最终回答。

## 5. 当前 Graph RAG 的工作流程

### 步骤 1：知识加载

系统从知识目录读取多格式文件，并附加结构化元数据。

### 步骤 2：向量入库

文档按规则切分后，生成 embedding 并写入本地向量索引。

### 步骤 3：图谱构建

根据元数据中的概念、别名和关系，构建本地图谱索引。

### 步骤 4：问题进入系统

用户提问后，系统先判断是否属于简单问候；若不是，则进入知识检索流程。

### 步骤 5：图谱检索

先根据问题中的关键词、概念、别名匹配图谱中的概念节点，得到：

- 命中概念
- 关联概念
- 关系边
- 推荐知识源

### 步骤 6：向量检索

再执行向量检索，从知识片段中找语义上最接近的问题候选。

### 步骤 7：混合排序

将图谱推荐结果作为 boost，叠加到向量检索结果上，得到最终候选片段。

### 步骤 8：候选过滤

通过关键词重叠、概念重叠、图谱命中情况判断片段是否真的相关。

### 步骤 9：用户单选

如果有命中的候选片段，前端让用户单选一条。

### 步骤 10：答案生成

模型基于：

- 用户问题
- 所选知识片段
- 图谱关系上下文

生成最终学习回答。

## 6. 这套技术方案的特点

### 优点

- 实现简单，易于落地
- 不依赖重型图数据库
- 相比纯向量 RAG，更适合教育知识场景
- 支持概念关系、前置依赖、章节信息、学科信息
- 便于后续继续升级为自动抽取图谱

### 当前限制

- 图谱关系目前主要来自手工元数据维护
- 还没有引入 reranker
- 还没有引入 BM25 等传统检索
- 还没有引入自动知识抽取与自动清洗流水线
- 当前图谱仍是轻量实现，适合中小规模知识库

一句话版本

- 我这个项目里的 RAG，本质上是一个 “文档解析 + 元数据增强 + 向量检索 + 图谱增强 + 大模型生成” 的知识问答系统。

面试标准回答

- 我先做了 知识入库 ，支持 PDF、DOCX、Markdown、TXT 多种格式，把原始资料统一解析成文本对象。
- 然后对文档做 切块 ，因为原始文档太长，不能直接整篇做检索。我用的是递归切分策略，控制 chunk 大小和 overlap，尽量保证语义完整。
- 切块之后，我用 embedding 模型把每个 chunk 转成向量，并把文本和 metadata 一起持久化到本地索引里。
- 查询时，先把用户问题也转成向量，做相似度召回，先拿到一批候选片段。
- 但我没有只依赖向量相似度，还加了 关键词重叠判断和元数据过滤 ，避免出现“语义相近但答案不准”的情况。
- 在这个基础上，我又扩展了一层 轻量 Graph RAG ：给知识源补了学科、章节、概念、别名、关系等结构化元数据，构建本地图谱索引。
- 查询时先从图谱里识别问题涉及的概念，再把图谱命中的关系和推荐来源用于提升检索排序，这样对于“概念关系、前置依赖、公式关联”类问题效果会更好。
- 最终我把用户问题、选中的知识片段、以及图谱补充关系一起喂给大模型，让它基于检索结果生成答案，而不是纯靠模型自由发挥。
如果面试官问“你具体怎么做入库的”

你可以这样答：

- 第二步是 元数据注入 ，我会给每个文档补上 source、file_type、subject、chapter、difficulty、concepts 这些字段。
- 第三步是 文本切块 ，我用了递归切分器，设置 chunk size 和 overlap，避免知识点被切断。
- 第四步是 向量化和持久化 ，调用 embedding 接口把 chunk 转成向量，用本地 numpy + json 存起来，形成一个轻量向量库。
- 同时我还会根据元数据构建一份 graph index，把概念节点、来源节点和关系边存成 JSON，给后续检索使用。
如果面试官问“检索阶段怎么做的”

你可以答：

- 检索阶段不是单一路径，而是 混合检索 。
- 一路是标准向量检索，先召回语义最接近的 chunk。
- 另一路是图谱检索，先识别问题命中的概念和别名，再找到相关关系边和高相关来源。
- 然后我把图谱结果转成 source boost，对向量召回结果做重排。
- 最后再用关键词重叠、概念重叠做一层过滤，只保留真正和问题相关的知识片段。
- 这样做的原因是，纯向量检索对“导数和极限的关系”“牛顿第二定律和加速度是什么关系”这种结构化问题不一定稳定，而图谱增强后效果更好。
如果面试官问“为什么不用现成的向量库或图数据库”

可以这样答：

- 这个项目一开始目标是快速验证教育场景下的知识问答效果，所以我优先做了一个 轻量、可控、低依赖 的实现。

- 如果后续数据规模上来，我会把向量层升级到更专业的向量数据库，把图谱层迁移到 Neo4j 或 NebulaGraph。
